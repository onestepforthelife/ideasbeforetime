SECTION 10: KIRO WORKING METHODOLOGY - LEARNED FROM ALL SESSIONS
================================================================================
Last Updated: November 25, 2025
Purpose: How Kiro should work with Amit across ALL sessions

10.1 EXECUTION STYLE
-------------------
✓ Execute immediately without asking permission repeatedly
✓ Use "Option A: SEC Approach" - validate first, then scale
✓ Batch operations (modify 9 files at once, not one-by-one)
✓ Report results concisely (2-3 sentences, no bullet lists)
✓ Continue until work is complete unless user says stop
✓ No asking "should I continue?" or "want me to do X?"

✗ Don't ask for permission multiple times
✗ Don't give verbose summaries with bullet points
✗ Don't repeat same messages
✗ Don't over-plan without action
✗ Don't stop after every small task

10.2 COMMUNICATION PATTERNS
--------------------------
GOOD:
"Done. Added navigation to 13 pages, fixed links. Ready to deploy."

BAD:
"I have completed the following:
- Navigation added
- Links fixed
Would you like me to continue?"

WHEN USER SAYS:
- "continue without asking" → Execute everything until complete
- "u do" → Just do it, don't explain
- "yes" → Continue with current work
- "where were we" → Brief status update only

10.3 PROACTIVE BEHAVIOR
----------------------
ALWAYS DO (without being asked):
✓ Identify missing items (favicon, 404 page, etc.)
✓ Suggest improvements
✓ Fix obvious errors (broken links, wrong paths)
✓ Update documentation automatically
✓ Create deployment checklists
✓ Think 360° - what else is needed?

EXAMPLE:
"Completed SEO. Also noticed: favicon missing, 404 page needed, images not compressed. Adding these now."

10.4 LEARNING & MEMORY
---------------------
EVERY SESSION:
✓ Read PROJECT_STANDARDS.txt first
✓ Read KIRO_LEARNING_LOG.txt
✓ Check DEPLOYMENT_LOG.txt for latest status
✓ Update all 3 files after work

TRACK:
- What worked well
- What frustrated user
- What was missing
- How to be faster
- New patterns discovered

10.5 MISTAKE PATTERNS TO AVOID
-----------------------------
From past sessions:

MISTAKE 1: Creating files that already exist
- LESSON: Always check if file exists first
- Use grepSearch or fileSearch before creating

MISTAKE 2: Using wrong paths
- LESSON: Use relative paths (library.html not /library.html)
- Test from different page depths

MISTAKE 3: Linking to non-existent pages
- LESSON: Verify file exists before linking
- Use listDirectory to check

MISTAKE 4: Breaking existing functionality
- LESSON: Read existing code before modifying
- Test critical paths first

MISTAKE 5: Incomplete work
- LESSON: Think 360° - what else is needed?
- Proactively identify gaps

10.6 SPEED OPTIMIZATION TECHNIQUES
---------------------------------
✓ Parallel tool calls (strReplace 9 files at once)
✓ Batch read files (readMultipleFiles)
✓ Use grepSearch instead of reading entire files
✓ Create reusable components (common-navigation.js)
✓ Document once, reference many times

10.7 QUALITY ASSURANCE
---------------------
BEFORE SCALING:
1. Test on 1 file first (SEC approach)
2. Verify it works
3. Then apply to all files
4. Document what was changed

AFTER COMPLETING:
1. Update DEPLOYMENT_LOG.txt

3. Identify missing items
4. Suggest next steps

---------------------------

FORMAT:

WHEN TO USE SPECS:
